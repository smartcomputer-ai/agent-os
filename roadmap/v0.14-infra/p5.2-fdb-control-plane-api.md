# P5.2: FDB Control Plane API (HTTP Reads, Ingress, Admin, CAS, Trace)

**Priority**: P5.2  
**Effort**: High  
**Risk if deferred**: High (hosted runtime exists but lacks a stable external ingress/admin surface)  
**Status**: Completed

Implementation target after P2/P3/P4 and P5.1:

- `crates/aos-fdb` already exposes the core hosted persistence/runtime/admin protocol for CAS, ingress, runtime views, world create, and world fork.
- `crates/aos-fdb-node` is the intended hosted process name for the combined worker/control node; the current implementation basis is the existing `aos-fdb-worker`.
- Missing now: a hosted API/control-plane process that exposes those capabilities as a stable external interface.

This milestone defines the full hosted HTTP control plane on top of the direct FDB-backed persistence plane and the small materialized read layer from P5.1.

Remaining direct-surface hardening follow-up work lives in P5.3:

- async delete and operation polling,
- smaller contract cleanup around consistency and optional raw journal encoding.

Persisted idempotency keys and CAS HTTP range reads are explicitly deferred beyond P5.3 for now.

## Goal

Ship the first hosted API/control plane for AgentOS universes/worlds so external clients can:

1. create and manage universes,
2. create/fork/list/read worlds,
3. send hosted ingress into worlds,
4. read hosted runtime, journal, and promoted head/query state,
5. inspect coarse persisted trace and runtime state for hosted worlds,
6. upload and read universe-scoped CAS content,
7. perform safe asynchronous destructive operations.

This is the API milestone, not a new runtime design. The worker/runtime semantics from P3 remain authoritative, and P5.1 is assumed for the promoted head/query reads.

## Why HTTP

Use HTTP as the primary protocol in v1.

Reasons:

- The hosted surface is operator/client facing, not an internal high-frequency mesh protocol.
- The primary operations are resource-oriented: create world, list worlds, enqueue event, read CAS blob, inspect workers, inspect journal.
- JSON plus binary bodies is sufficient and keeps tooling friction low.
- Existing local HTTP work in `aos-cli` already shows the repo is comfortable with Axum-style JSON APIs.
- gRPC would add complexity before we need streaming RPC semantics, generated clients, or strict service-mesh integration.

Design stance:

- HTTP/1.1 and HTTP/2 should both work.
- JSON is the default envelope for metadata and control requests.
- Raw request/response bodies should be used for CAS blob upload/download.
- OpenAPI should be generated from the server crate for discoverability.

Non-decision:

- This does not prohibit a later gRPC facade for internal or multi-language consumers.
- HTTP is simply the required and sufficient first control-plane surface.

- Requires `v0.20-infra/p2-hosted-persistence-plane.md` merged.
- Requires `v0.20-infra/p3-universe-runtime-plane.md` merged.

P5 assumes the following are already true:

- hosted ingress is durable and queue-backed,
- workers own lease-controlled journal mutation,
- world create/fork from uploaded replay roots already works,
- universe-scoped CAS is authoritative for immutable bodies.

## Non-Goals (P5)

- Public internet hardening or full production auth stack.
- Tenant billing/quota enforcement.
- A second authority path around worker leases.
- Replacing worker-driven runtime scheduling with synchronous API-driven execution.
- A websocket-first or streaming-first operator protocol.
- Full mark-and-sweep deletion execution and CAS GC policy engine.
- Embedded/local runtime API parity.
- Recreating the old `aos-cli` daemon API exactly.

## Design Stance

- Do not add a separate permanent control-plane crate.
- The hosted API/control plane should live in the hosted FDB process crate alongside the worker runtime.
- The control plane talks directly to `aos-fdb` protocol traits; it does not speak to workers over an ad hoc side channel.
- The control plane is authoritative for admin-plane mutations and ingress enqueue only.
- Workers remain authoritative for lease-guarded runtime mutations.
- Public API shapes should be resource-oriented and HTTP-native, not thin exposure of every trait method.
- The API should preserve current hosted invariants instead of hiding them.

Required separation:

- `HostedWorldAdminPersistence` and future universe-admin traits back admin endpoints.
- `HostedRuntimePersistence` backs runtime read views and ingress enqueue endpoints.
- Guarded lease mutation APIs remain internal-only.

Process model:

- The hosted process should support composable roles, not separate permanent binaries.
- Minimum roles:
  - `worker`
  - `control`
  - `worker,control`
- A single process must be allowed to run both worker and control roles at the same time.
- Running control-only should also remain possible for deployments that want to split ingress/admin from execution without changing the codebase shape.

Naming decision:

- The hosted multi-role process is named `aos-fdb-node`.
- `aos-fdb-worker` is now transitional terminology from P3 and the current implementation base, not the intended end-state name.
- P5 assumes a node-style process that may run worker and control roles together or separately.

## Resource Model

### 1) Universe

A universe is the top-level hosted isolation boundary:

- universe-scoped CAS,
- universe-scoped worker/runtime metadata,
- universe-scoped world catalog.

P5 introduces explicit universe admin records and APIs. P2/P3 operate on `UniverseId`, but do not yet define a universe catalog/admin surface.

Suggested universe metadata:

```text
UniverseRecord {
  universe_id: Uuid,
  created_at_ns: u64,
  display_name?: String,
  labels: map<string,string>,
  status: "active" | "deleting",
  deletion?: DeletionStatus
}
```

Suggested trait addition:

```rust
pub trait HostedUniverseAdminPersistence: Send + Sync {
    fn create_universe(&self, request: CreateUniverseRequest)
        -> Result<UniverseCreateResult, PersistError>;

    fn get_universe(&self, universe: UniverseId)
        -> Result<UniverseRecord, PersistError>;

    fn list_universes(&self, limit: u32, cursor: Option<String>)
        -> Result<Page<UniverseRecord>, PersistError>;

    fn mark_universe_deleting(&self, universe: UniverseId, request: DeleteUniverseRequest)
        -> Result<DeleteOperation, PersistError>;
}
```

Universe creation semantics:

- `POST /v1/universes` allocates a new `UniverseId` unless caller supplies one.
- It creates metadata only; it does not implicitly create a world.

Universe deletion semantics:

- `DELETE /v1/universes/{id}` is asynchronous.
- Initial implementation should mark the universe as `deleting`, reject new ingress/admin mutations, and return an operation handle.
- Physical purge is follow-on work.

### 2) World

World endpoints expose the hosted world admin model already present in `aos-fdb`.

P5 should extend hosted world metadata with an explicit admin status field rather than treating deletion as an API-only concern.

Suggested addition:

```text
WorldAdminStatus = "active" | "deleting"
```

This can live either as a new field on `WorldMeta` or as a closely-associated admin record, but it must be persisted and visible to:

- control-plane reads,
- ingress admission checks,
- lease acquisition eligibility checks,
- delete operation progress tracking.

`POST /v1/universes/{u}/worlds` should create from seed:

- request body mirrors `CreateWorldRequest`,
- seed references already-uploaded manifest and snapshot CAS roots,
- response returns the resulting `HostedWorldRecord`.

`POST /v1/universes/{u}/worlds/{w}/fork` should fork:

- request body mirrors `ForkWorldRequest`,
- response returns the resulting `HostedWorldRecord`.

`GET /v1/universes/{u}/worlds/{w}` should return:

- world metadata,
- active baseline summary,
- journal head,
- runtime readiness summary,
- lease summary if present.

`PATCH /v1/universes/{u}/worlds/{w}` should initially support:

- `placement_pin`,
- optional display metadata/labels if we add them,
- future admin toggles such as `deleting`.

World deletion semantics:

- `DELETE /v1/universes/{u}/worlds/{w}` must be asynchronous.
- First phase should mark the world `deleting`, reject new ingress, and ensure it is no longer eligible for new lease acquisition.
- If a worker still holds a lease, deletion waits for release or expiry.
- Hard purge of journal/snapshots/CAS reachability is out of scope for P5.

### 3) Ingress

Ingress remains durable enqueue, not synchronous execution.

`POST /events`:

- accepts a typed domain event request,
- validates basic shape,
- externalizes large payloads to CAS if needed,
- enqueues `InboxItem::DomainEvent`,
- returns `202 Accepted` with `inbox_seq`.

`POST /receipts`:

- accepts a `ReceiptIngress`-shaped body,
- enqueues `InboxItem::Receipt`,
- returns `202 Accepted` with `inbox_seq`.

This preserves the P3 rule:

- all external inputs converge into durable inbox,
- only the lease holder drains inbox to journal and advances runtime.

The API must not:

- append directly to journal,
- force a worker to run immediately,
- bypass lease fencing.

### 4) Runtime, Journal, Promoted Reads, And Observability

Read APIs should expose hosted state as direct FDB-backed reads or FDB-backed projections, without requiring restored-world execution.

Required reads:

- `GET /manifest`,
- `GET /defs`,
- `GET /defs/{kind}/{name}`,
- `GET /workspace/resolve`,
- `GET /state/{workflow}`,
- `GET /state/{workflow}/cells`,
- `GET /runtime` for `WorldRuntimeInfo`,
- `GET /journal/head`,
- `GET /journal?from=&limit=`,
- `GET /trace`,
- `GET /trace-summary`,
- `GET /workers`,
- `GET /workers/{worker_id}/worlds`.

Journal read response should be paginated by journal height:

```text
{
  "from": 42,
  "next_from": 142,
  "entries": [...]
}
```

The control plane may decode journal entries to JSON for convenience, but it should also support a raw CBOR mode later if needed.

World reads should include active baseline metadata in either:

- `GET /worlds/{id}`, or
- `GET /worlds/{id}/runtime`,

so a separate public `snapshots/active` route is not required in v1.

Promoted read rules:

- current `manifest`, `defs`, and `def-get` are served from the current head projection plus CAS,
- `workspace/resolve` is served from the workspace registry projection,
- `state-get` and `state-list` are served from the latest-durable cell-state projection plus CAS,
- no restored-world read path is required for the main P5.2 read surface.

Hosted observability rules:

- `trace` is a hosted persisted-state trace, not a clone of the old daemon `trace-get` response,
- it should be built from durable FDB-backed sources such as journal, inbox/runtime state, and persisted effect/timer state,
- it may include a journal window, runtime summary, inbox/effect/timer summaries, and coarse terminal status,
- it should not depend on hot in-memory worker state or restored-world inspection,
- `trace-summary` should provide the compact operator view that the existing CLI surface relies on.

Read consistency model:

- `latest_durable`
- `head`
- `exact:{height}`
- `at_least:{height}`

Initial operational meaning:

- `latest_durable` is the primary direct-read mode and is backed by projections,
- `head` maps to `latest_durable` in the initial hosted implementation,
- `exact:{height}` and `at_least:{height}` are part of the API contract, but routes may initially reject combinations that are not yet backed by direct historical lookups,
- `latest_live` is intentionally not part of P5.2.

Deferred from the main public v1 surface:

- `GET /snapshots/{height}`,
- `GET /segments`,
- exact parity with the local daemon `trace-get` response.

### 5) Direct Workspace / Tree APIs By Root Hash

The CLI already relies heavily on workspace/tree operations, but not all of them require current running world state.

P5 should expose a smaller tree API that can be served directly from immutable roots and CAS without restoring a world:

- `POST /workspace/roots`
- `GET /workspace/roots/{root_hash}/entries`
- `GET /workspace/roots/{root_hash}/entry`
- `GET /workspace/roots/{root_hash}/bytes`
- `GET /workspace/roots/{root_hash}/annotations`
- `POST /workspace/roots/{root_hash}/apply`
- `POST /workspace/diffs`

This should follow the same broad model as the composite `workspace_apply` tool already used in `aos-agent`, but adapted to an HTTP/root-oriented control plane:

- `resolve` stays a distinct read operation,
- `apply` runs an ordered list of tree mutations against an explicit base root,
- `commit` is intentionally separate and remains a later authoritative world-write concern.

Directness rule:

- if the request is anchored by explicit root hash inputs, the node should answer directly from tree/CAS state,
- if the request depends on named workspace resolution against current world state, it is not part of the raw root-hash API surface and belongs in the projection/read-model layer from P5.1.

This intentionally separates:

- root-addressed tree manipulation,
- from named workspace resolution and commit flows.

Suggested request model:

- `entries` lists directory/tree entries under an optional `path` and may support `scope`, pagination, and cursoring,
- `entry` returns metadata for a single path, including whether it is a tree node, blob ref, mode, size, and available annotations metadata,
- `bytes` returns raw bytes for a file path and may support ranges,
- `annotations` reads the annotation map for a path,
- `roots/{root_hash}/apply` accepts an ordered patch of tree mutations and returns the resulting root hash,
- `workspace/diffs` diffs two explicit roots or two explicit root+path scopes.

The main abstraction change from the old daemon API is that write operations should be batched under one patch/apply endpoint instead of being split into many tiny verbs.

Conceptually:

1. `workspace/resolve` answers "which root should I start from?"
2. `workspace/roots/{root_hash}/apply` answers "what new root do I get after these tree edits?"
3. a later workspace commit/control-mailbox API answers "make this new root authoritative for workspace X"

This intentionally keeps immutable tree transformation separate from world mutation.

Suggested `apply` operation kinds:

- `write_bytes`
- `write_ref`
- `remove`
- `set_annotations`

This stays low-level enough to expose the immutable tree substrate directly, but avoids overfitting the public API to the internal control verbs used by the CLI.

Suggested `apply` request shape:

```text
WorkspaceApplyRequest {
  operations: [
    {
      op: "write_bytes",
      path: "src/main.rs",
      bytes_b64: "..."
      mode?: 33188
    },
    {
      op: "write_ref",
      path: "assets/logo.png",
      blob_hash: "sha256:..."
      mode?: 33188
    },
    {
      op: "remove",
      path: "old.txt"
    },
    {
      op: "set_annotations",
      path: "README.md",
      annotations_patch: { "owner": "sha256:...", "reviewed": null }
    }
  ]
}
```

Semantics:

- operations are applied in order,
- each operation sees the root produced by the previous operation,
- the response returns `base_root_hash`, `new_root_hash`, and an optional compact `changes` summary,
- blob bytes should be uploaded through CAS first and then referenced by `write_ref` when the client already has a stable blob hash,
- a convenience `write_bytes` op remains worthwhile because it avoids forcing the client to manually split tiny file writes into `cas put` + `write_ref`.

Relation to the existing `aos-agent` composite tool:

- same high-level idea: one request, ordered operations, one resulting root,
- different public boundary: HTTP `apply` should stay root-based and pure,
- no implicit named-workspace commit inside `apply`,
- no need to expose the internal sequence of lower-level `workspace.write_*` effects in the public API.

Named workspace write APIs intentionally deferred beyond P5 direct surface:

- workspace commit against current world state,
- `push` / `pull` style sync workflows anchored in the world’s active workspace registry.

Follow-on split:

- `workspace-resolve` is part of the P5.2 read API and should use the P5.1 workspace registry projection,
- workspace commit and sync flows remain authoritative world-write concerns for P6.

### 6) Governance Deferral

Governance does not belong to the direct FDB-backed P5.2 surface.

Rules:

- governance propose/shadow/approve/apply move to P6,
- those routes execute through the world control mailbox, not as direct FDB mutations,
- P5.2 does not need governance endpoints to ship the direct hosted read/control surface,
- `gov-apply-direct` remains intentionally out of the hosted public API.

### 7) CAS

CAS should use HTTP-native binary transfer.

Required routes:

- `POST /cas/blobs`
  - body: raw bytes
  - server computes SHA-256
  - response: hash and metadata
- `PUT /cas/blobs/{sha256}`
  - body: raw bytes
  - server verifies hash matches path
  - useful for idempotent clients and precomputed hashes
- `HEAD /cas/blobs/{sha256}`
  - existence and metadata check
- `GET /cas/blobs/{sha256}`
  - returns raw bytes
  - should support HTTP `Range`

Rules:

- Do not wrap blob transport in base64 by default.
- Universe stays in the URL path so CAS remains explicitly universe-scoped.
- Large request handling may stream internally, but the initial external contract need not expose multipart upload orchestration.

## Operations Model

Any mutation that may require waiting, quiescence, or multi-phase cleanup should return an operation.

Required operation-backed actions:

- universe delete,
- world delete,
- future import/export jobs,
- future GC or maintenance jobs if operator-triggered.

Suggested shape:

```text
Operation {
  operation_id: String,
  kind: String,
  status: "pending" | "running" | "succeeded" | "failed",
  submitted_at_ns: u64,
  started_at_ns?: u64,
  finished_at_ns?: u64,
  resource?: String,
  error?: ApiErrorBody
}
```

HTTP behavior:

- `202 Accepted` with operation body for async mutations.
- `GET /v1/operations/{id}` to poll.

## Request/Response Conventions

### JSON

Use JSON for all control-plane request and response metadata.

Field conventions:

- snake_case field names,
- UUIDs and hashes rendered as strings,
- explicit status enums rather than booleans where state may expand,
- `meta` blocks only where they add real value.

### Binary

Use raw bytes for CAS upload/download.

Headers to support:

- `Content-Type: application/octet-stream`
- `Content-Length`
- `ETag` for CAS hash
- `Range` / `Content-Range`

### Idempotency

Mutating endpoints should accept `Idempotency-Key`.

Required initially for:

- `POST /universes`
- `POST /worlds`
- `POST /worlds/{id}/fork`
- `POST /events`
- `POST /receipts`
- `DELETE` endpoints

Semantics:

- repeated requests with same key and equivalent body return the same logical result,
- mismatched body under the same key is an error,
- implementation may persist idempotency windows in FDB.

## Error Model

The API should translate persistence/runtime conditions into explicit HTTP statuses.

Suggested mapping:

- `400 Bad Request`
  - malformed JSON,
  - invalid schema names,
  - invalid hash/path mismatch,
  - impossible request combinations
- `404 Not Found`
  - universe/world/blob absent
- `409 Conflict`
  - world already exists,
  - universe already exists,
  - delete blocked by incompatible current state,
  - idempotency key mismatch
- `422 Unprocessable Entity`
  - referenced manifest/snapshot roots missing,
  - seed/fork validation failures
- `423 Locked`
  - optional for destructive operations blocked by active lease if we want that distinction
- `500 Internal Server Error`
  - unexpected backend failure
- `503 Service Unavailable`
  - startup/dependency unavailable

Suggested error body:

```text
{
  "code": "world_exists",
  "message": "world 2222... already exists",
  "details": { ... }
}
```

Do not leak raw FoundationDB client details in public responses.

## Auth and Policy Staging

P5 should define an API surface that can later be hardened, but auth itself remains minimal for the first internal rollout.

Recommended first phase:

- bind on internal network only,
- optional static bearer token,
- structured request logging,
- no anonymous public exposure.

Follow-on phases may add:

- per-universe RBAC,
- scoped API keys,
- mTLS for service-to-service usage,
- audit actor attribution on admin mutations.

Important boundary:

- API auth is not a substitute for runtime capability/policy enforcement inside worlds.
- It governs who may call the hosted control plane, not what a workflow may emit once running.

## Relation To Existing `aos-cli` HTTP API

Use `aos-cli` HTTP only as ergonomic inspiration.

Keep:

- Axum-style implementation,
- OpenAPI generation,
- straightforward JSON request/response patterns,
- separate health/info/read/write route families.

Do not keep:

- world-local filesystem assumptions,
- daemon-control command tunneling,
- exact route names or payload compatibility,
- daemon lifecycle operations such as `run`, `stop`, `shutdown`, `snapshot create`, or `journal replay`,
- client workflow commands such as `push` and `pull`,
- server-side `trace-find` or `trace-diagnose` endpoints in v1; `journal` and coarse `trace` should remain the substrate,
- full debug parity where the old daemon depended on live in-memory world state.

Hosted control plane and local daemon API serve different jobs.

## Operational Notes

- The control plane should be stateless aside from its use of FDB.
- Horizontal scale should be safe as long as idempotency and operation records are persisted.
- API timeouts should remain modest; long work should become operations rather than held requests.
- Request/response logging should include `universe_id`, `world_id`, `operation_id`, and idempotency key where present.

## Open Questions

Questions to resolve during implementation:

1. Should universe creation optionally create a default world, or remain metadata-only?
2. Do we want `423 Locked` for active-lease delete blockers, or fold everything into `409 Conflict`?
3. Should journal reads expose decoded JSON only, or support `application/cbor` from the start?
4. Do we want world labels/display metadata in P5, or keep world metadata minimal?
5. Should idempotency records be universe-scoped only, or endpoint-scoped within a universe?
6. How much persisted effect/timer runtime state do we need to expose through `aos-fdb` to make hosted `trace` and `trace-summary` useful without restoring a world?

## Exit Criteria

P5 is complete when:

1. A hosted HTTP service can create a universe.
2. A client can upload manifest/snapshot blobs into universe CAS.
3. A client can create or fork a world from those roots.
4. A client can enqueue a domain event or receipt into a world through HTTP.
5. A client can read current manifest, defs, workspace resolve, and latest-durable state through HTTP without restored-world execution.
6. A client can inspect world runtime, journal, worker placement, and coarse hosted trace/debug state through HTTP.
7. Governance-style authoritative world writes are deferred to P6 and do not block P5.2 completion.
8. A client can start logical delete for a world or universe and poll the operation.
9. The API preserves existing hosted invariants: inbox-first ingress, worker-owned journal mutation, lease fencing, and universe-scoped CAS.
