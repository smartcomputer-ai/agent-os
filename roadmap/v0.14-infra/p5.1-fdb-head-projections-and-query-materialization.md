# P5.1: FDB Head Projections And Query Materialization

**Priority**: P5.1  
**Effort**: Medium  
**Risk if deferred**: Medium-High (too many high-value reads remain artificially classified as restored-world operations)  
**Status**: Completed

## Goal

Introduce the tiny hosted projection/materialization layer that prepares selected high-value reads to be served cheaply from FDB-backed metadata later in P5.2.

This milestone exists to keep the stack clean:

1. small materialized head/query projections in P5.1,
2. direct FDB-backed control-plane APIs in P5.2,
3. authoritative world writes through the control mailbox in P6.

Without P5.1, too many important reads get forced into an unnecessary "restore the world" bucket even when the answer can be made cheap with a very small durable projection.

## Design Stance

- Keep projections minimal and targeted.
- Only project data that materially reduces restore-backed read traffic.
- Prefer immutable or monotonic projections where possible.
- Do not build a generic secondary-index framework.
- Do not create a second source of truth for world execution.

Projection rule:

- the world engine remains authoritative,
- projections are derived materialized views for hosted query efficiency,
- projections exist to prepare the later read/control plane; they are not themselves an API milestone.

## Why P5.1 Exists

Some queries currently look like restored-world reads only because the hosted persistence plane does not yet surface a tiny piece of current-head metadata.

The clearest example is current manifest access:

- historical manifest reads are already direct once the manifest hash is known,
- therefore current-head manifest/defs reads become cheap once we surface the current head manifest hash durably.

That same idea likely applies to a few more things:

- named workspace registry lookup,
- latest durable cell-state lookup,
- small trace summary projections,
- other very high-value head metadata that does not require full interpreted state.

## Projection Families

### 1) Head Manifest Projection

Persist a direct reference to the world’s current head manifest hash.

Suggested shape:

```text
HeadProjection {
  journal_head: u64,
  manifest_hash: String,
  updated_at_ns: u64
}
```

This may live as:

- a field on durable runtime metadata,
- a dedicated per-world projection record,
- or another small persisted record in `aos-fdb`.

Requirements:

- updated atomically with authoritative manifest-changing world commits,
- monotonic in journal height,
- cheap to read with world metadata.

### 2) Active Workspace Registry Projection

If hosted workspace resolution is a first-class hot path, project the active workspace registry.

Suggested shape:

```text
WorkspaceRegistryProjection {
  journal_head: u64,
  workspaces: map<workspace_name, WorkspaceBinding>
}
```

Scope should stay intentionally narrow:

- enough to resolve current workspace name/version/root,
- not a full duplicate of all workspace tree content.

### 3) Optional Small Trace / Shell Summary Projections

Only add these if needed after real usage.

Candidates:

- last failure summary,
- last N command results,
- compact workflow activity summaries.

These are not required for the initial P5.5 milestone.

### 4) Latest-Durable Cell State Projection

All state reads are effectively cell reads in the current kernel model:

- keyed workflows read a specific keyed cell,
- non-keyed workflows read the sentinel `MONO_KEY` cell.

That makes cell state a good candidate for the same "slightly more metadata" approach, with one crucial limitation:

- the projection can only represent the latest durable state,
- not the latest hot in-memory state on an active worker if writeback is deferred for performance.

Suggested projection shape:

```text
CellStateProjection {
  journal_head: u64,
  workflow: String,
  key_hash: bytes,
  key_bytes?: bytes,
  state_hash: String,
  size: u64,
  last_active_ns: u64
}
```

Design decision:

- do not inline state bytes into projection rows,
- keep projections metadata-only and fetch actual state bytes from CAS when later read APIs need them.

Semantics must be explicit:

- the projected answer is `latest_durable`,
- not `latest_live`.

This distinction matters because we may intentionally avoid writing every hot worker cell-state change back to CAS/index/metadata immediately for performance reasons.

## Relation To P5.2, P6, And P7

P5.1 should make the main hosted read surface cheap enough that P5.2 can stay purely FDB-backed:

- current `manifest`,
- current `defs`,
- current `def-get`,
- current `workspace-resolve` if projected,
- `latest_durable` state-get / state-list if projected.

P6 should then focus only on authoritative world writes through the control mailbox.

Anything that still truly needs restored interpreted world state becomes a later follow-on problem, not part of the main hosted control plane:

- state reads that need more than `latest_durable`,
- shell inspection,
- richer trace/diagnostics,
- other derived runtime views not cheaply projectable.

Those future advanced/live reads belong conceptually with P7, not with the main P5.2/P6 control-plane plan.

## Scope (Now)

### [x] 1) Add head manifest projection

Implement a small durable current-head manifest hash projection in `aos-fdb`.

Completed:

- durable `HeadProjection` records exist in `aos-fdb`,
- seeded from baseline snapshots,
- refreshed by workers as part of authoritative projection materialization,
- readable through the hosted persistence Rust API.

### [x] 2) Expose Rust persistence reads needed by P5.2

Completed:

- projection records are readable through the hosted persistence Rust API,
- `head_projection` is available for current-head manifest/defs reads,
- workspace registry projection reads are available for named workspace resolution,
- cell-state projection reads are available for `latest_durable` state reads.

### [x] 3) Evaluate and optionally add active workspace registry projection

If workspace name resolution is important enough, add the projection and promote `workspace-resolve` too.

Completed:

- active workspace registry projection was implemented,
- versioned workspace bindings are persisted durably,
- projections are seeded from baseline snapshots and refreshed by workers,
- multiple workspace versions can be resolved later from durable metadata.

### [x] 4) Add latest-durable cell state projection

Implement:

- latest durable state metadata per `(workflow, key)`,
- metadata needed for later `latest_durable` state-get / state-list,
- no inline state duplication in projection rows.

Completed so far:

- durable latest-durable cell projection records exist,
- per-cell projection updates are incremental in the persistence/runtime layer,
- the Rust hosted persistence API can already read projected state metadata.

### [x] 5) Define non-goals for initial hosted read semantics

The initial hosted read surface intentionally does not implement:

- `latest_live` state reads,
- shell-style read sessions,
- richer trace/diagnostic queries that need restored interpreted world state.

Decision made:

- `latest_live` remains out of scope for this milestone,
- advanced restored/live read paths remain deferred beyond P5.1/P5.2.

## Exit Criteria

P5.1 is complete when:

1. Durable current-head manifest projection exists and is maintained authoritatively.
2. Durable workspace registry projection exists and is maintained authoritatively.
3. Durable latest-durable cell-state projection exists and is maintained authoritatively.
4. The hosted persistence Rust API can read all projection families needed by P5.2.
5. P5.2 can expose the main hosted read surface without introducing new restore-backed read requirements.

## Current Status

Completed in storage/runtime:

- durable head manifest projection
- durable workspace registry projection with version bindings
- durable latest-durable cell state projection
- worker-driven projection refresh
- incremental per-cell / per-workspace projection writes
- Rust hosted persistence read APIs for projections

Still left before P5.1 can be considered fully complete:

- no additional storage/runtime projection families are required for the planned main read surface,
- completion now depends mostly on P5.2 consuming this projection substrate cleanly,
- optional extra promoted read models should only be added if P5.2 reveals a real gap.
