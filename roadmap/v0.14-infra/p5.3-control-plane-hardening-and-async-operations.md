# P5.3: Control Plane Hardening And Async Operations

**Priority**: P5.3  
**Effort**: Medium  
**Risk if deferred**: Medium (the direct hosted control plane works, but some important operational contracts and destructive flows are still incomplete)  
**Status**: Done

## Current Status

As of 2026-03-11, the remaining scope from this milestone is implemented.

- [x] World-scoped async lifecycle submission and polling now exist via the P6 command mailbox.
- [x] Universe delete now exists as a guarded direct metadata transition and only succeeds once every world in the universe is already marked `deleted`.
- [x] Durable status records exist for those world-scoped requests (`CommandRecord`), and polling is a pure read through `GET /v1/universes/{universe_id}/worlds/{world_id}/commands/{command_id}`.
- [x] Unsupported projected-state consistency values are rejected explicitly; only `latest_durable` and `head` are accepted on the shipped projected state reads.
- [x] Optional raw journal CBOR encoding now exists on the hosted journal endpoint.

Important clarification:

- P6 intentionally standardized world-scoped leased execution on `command_id` / `CommandRecord`, not on a second generic `operation_id` abstraction.
- P5.3 now follows that same direction instead of keeping a separate generic `operations` resource.
- Universe delete remains a direct hosted metadata action; world-scoped lifecycle work remains leased command execution.

## Goal

Finish the remaining direct HTTP/FDB control-plane concerns that were intentionally left out of the initial P5.2 implementation.

This milestone is narrow:

1. P5.1 prepared the durable read substrate,
2. P5.2 shipped the main direct hosted control surface,
3. P5.3 hardens that surface for operations and a few remaining direct HTTP contracts,
4. P6 owns leased execution of world-scoped lifecycle commands through the control mailbox.

## Why P5.3 Exists

P5.2 is enough to:

- create universes and worlds,
- upload blobs,
- enqueue ingress,
- read runtime, journal, projections, and coarse trace,
- use direct workspace/tree operations by root hash.

But a few important direct-surface pieces are still missing:

- async delete and operation polling,
- a couple of API contract follow-ups around consistency and encoding.

These do not justify reopening P5.2 or blocking P6, but they are still real work.

## Design Stance

- Keep P5.3 focused on the operation API and direct control-plane contracts.
- Do not execute world-scoped lifecycle transitions directly in P5.3.
- Prefer explicit operation records over long-held HTTP requests.
- Treat retry/idempotency semantics as persisted control-plane state, not in-memory behavior.
- Keep the API resource model stable; this is hardening, not redesign.

## Scope

### 1) Async Lifecycle Operation API

Implement the remaining lifecycle/admin delete surface that P5.2 still left open:

- `DELETE /v1/universes/{universe_id}`
- `DELETE /v1/universes/{universe_id}/worlds/{world_id}`

Initial scope:

1. accept lifecycle requests through the hosted control plane,
2. submit world-scoped lifecycle requests to the leased execution path from P6,
3. allow guarded universe deletion once every world is already marked deleted,
4. leave physical purge and long-term retention as follow-on work.

Lifecycle families in scope:

- delete world
- pause world
- archive world
- later universe-scoped variants where appropriate

Why this belongs in P5.3:

- the external HTTP lifecycle/admin contract belongs to the control plane,
- but actual world-scoped lifecycle execution should happen under lease in P6,
- so P5.3 provides the durable operator-facing shell around those transitions.

Current implementation status:

- [x] World-scoped lifecycle submission is shipped through `POST /pause`, `POST /archive`, and `DELETE /worlds/{world_id}`.
- [x] Those routes create durable world-scoped command records and hand execution to the leased worker path from P6.
- [x] Polling for those world-scoped requests is shipped through `GET /v1/universes/{universe_id}/worlds/{world_id}/commands/{command_id}`.
- [x] The universe-scoped delete route (`DELETE /v1/universes/{universe_id}`) is shipped as a guarded direct metadata transition.
- [x] There is intentionally no separate persisted generic `operation` record model; the durable async status model is `CommandRecord` for world-scoped commands.

### 2) Consistency Contract Cleanup

P5.2 currently exposes the read consistency vocabulary:

- `latest_durable`
- `head`
- `exact:{height}`
- `at_least:{height}`

But only the first two are meaningfully implemented today.

P5.3 should decide and document one of these paths:

- explicitly reject unsupported `exact` / `at_least` combinations with stable errors, or
- implement the small subset that can already be answered directly without restored-world execution.

This is mostly contract hardening, not a large new feature.

Current implementation status:

- [x] Projected state reads reject unsupported values explicitly with stable `invalid_request` errors.
- [x] The shipped projected read surface accepts `latest_durable` / `head` and rejects everything else rather than silently pretending to support `exact:{height}` or `at_least:{height}`.
- [ ] Add explicit HTTP coverage for rejected `exact:{height}` / `at_least:{height}` inputs if we want the original test matrix fully closed.

### 3) Optional Raw Journal Encoding

The direct journal endpoint currently returns decoded JSON only.

P5.3 may add:

- optional `application/cbor` or equivalent raw mode for journal reads,
- while keeping the existing JSON convenience form.

This is lower priority than the operations API, but it is the last notable P5.2 contract gap.

Current implementation status:

- [x] Implemented. The hosted journal endpoint now supports raw CBOR mode alongside the existing decoded JSON form.

## Explicit Deferrals

The following items are intentionally deferred beyond P5.3:

### Idempotency-Key Persistence

Persisted `Idempotency-Key` handling is useful, but not required to finish the current direct control-plane follow-up slice.

Reasons to defer:

- the direct hosted surface already works correctly without it,
- CAS writes are naturally close to idempotent because they are content-addressed,
- the most urgent remaining gap is delete/operation handling, not retry replay semantics.

If/when we add it later, it should cover:

- universe/world create,
- ingress endpoints,
- delete endpoints.

### CAS Range Reads

HTTP range support for CAS reads is also deferred.

Reasons to defer:

- whole-blob GET already works,
- current direct-control-plane functionality does not depend on partial reads,
- this can be added later without changing the resource model.

If/when we add it later, it should support:

- `Range`
- `Content-Range`
- `206 Partial Content`
- `416 Range Not Satisfiable`

## Non-Goals

- governance writes
- workspace commit
- push/pull style world mutations
- leased execution of pause/archive/delete transitions
- restored-world read paths
- latest-live reads
- rich/live trace parity with the old daemon

Those belong to P6 or later milestones, not P5.3.

## Exit Criteria

P5.3 is complete when:

- [x] The hosted API can already start pause/archive/delete-style world commands and return durable async IDs (`command_id`).
- [x] Polling for those world-scoped async commands is a pure read and does not finalize work.
- [x] World-scoped lifecycle work is handed off to the leased execution model from P6.
- [x] The hosted API can delete a universe once all of its worlds are already marked deleted.
- [x] Unsupported projected-read consistency combinations are handled explicitly.
- [x] The generic `operations` resource has been removed from this milestone in favor of the shipped `commands` model plus guarded direct universe delete.
- [x] The remaining direct control-plane gap around optional raw journal encoding is closed.
