# P2.1: Hosted Manifest/Baseline Consistency

**Priority**: P2.1  
**Effort**: Medium  
**Risk if deferred**: High (a world can become logically inconsistent even when the underlying storage is intact)  
**Status**: Complete

## Goal

Close the remaining correctness gap between:

- journaled manifest history,
- indexed snapshots,
- active baseline promotion,
- hosted repair/recovery tooling.

This note exists because we hit a real hosted-world failure mode where startup could be made to "work" by changing the baseline manifest hash, but the resulting world no longer matched its own journal history.

The storage itself was not corrupt at the byte level. The corruption was semantic: the baseline stopped describing the same world that the journal described.

## Current Status

As of 2026-03-13, the core guardrail from this note is implemented:

- snapshot restore now validates snapshot manifest lineage against journaled manifest history,
- explicit baseline promotion applies the same validation,
- mismatched snapshot seeds fail closed instead of being trusted as replay baselines,
- the ad hoc hosted repair path that allowed arbitrary manifest stamping has been removed from `hosted_probe`.

This closes the specific corruption path that produced bootable-but-inconsistent hosted worlds.

## What Failed

The hosted Demiurge world ended up with two conflicting truths:

1. the journal history said the world had evolved through a particular sequence of manifest records, and
2. the repaired active baseline/snapshot state pointed at a different manifest hash.

That mismatch is enough to make replay and runtime behavior incoherent:

- workflow state may be restored under the wrong module WASM,
- receipt continuation may target a workflow implementation that did not create the original pending context,
- schema/event decoding can become invalid at replay time,
- hosted queries can appear mostly healthy while live execution is subtly wrong.

In our case, recovery attempts made the world bootable, but not historically self-consistent.

## What Happened

The sequence was roughly:

1. The world had a real journaled manifest history.
2. Historical hosted snapshots and baseline promotion behavior had gaps:
   - latest replay snapshots were not always promoted cleanly,
   - legacy snapshot rows could carry incomplete metadata,
   - stale runtime bookkeeping could strand maintenance.
3. During recovery, the active baseline/runtime manifest hash was effectively rewritten to a manifest that allowed startup.
4. That manifest was not the same manifest the journal said the world should be on at that point.
5. Replay then operated across a baseline/journal split-brain:
   - snapshot state from one manifest lineage,
   - journal tail from another.

That is the core corruption class.

## Why This Was Possible

We had resilience mechanisms, but not a hard consistency invariant.

Missing guarantees:

- Snapshot manifest hash was not treated as a strictly derived value from journal history.
- Startup/open did not fail closed on baseline/journal manifest mismatch.
- Recovery tooling could make a world boot again without proving that the repaired baseline matched the journal at the same height.
- We lacked an explicit invariant check of the form:
  - "the snapshot/baseline manifest hash at height `H` must equal the manifest obtained by replaying manifest records through `H`."

In short: we had repair knobs, but not enough proof obligations around them.

## What We Fixed Already

Several real hosted runtime issues are fixed now, and they materially reduce the chance of drifting worlds:

### 1) Snapshot Healing and Promotion

- legacy indexed snapshots are healed on open,
- latest valid replay snapshots can now be promoted correctly,
- active baseline advancement is no longer stranded behind broken metadata.

This fixes the "stuck on height 1" class of failures.

### 2) Replay Seed Selection

- hosted reopen now uses the latest valid replay snapshot rather than accidentally biasing to ancient baseline state,
- `snapshot_latest()` returns the highest indexed snapshot correctly.

This fixes the catastrophic replay-cost path and avoids reopening from obviously stale seeds.

### 3) Stale Runtime Bookkeeping Cleanup

- stale pending receipt/intention bookkeeping is now pruned,
- maintenance can heal and persist a clean snapshot instead of looping forever on dead waits.

This fixes one class of false "world still has work in flight" states.

### 4) Better Diagnostics

Replay/startup failures now report enough context to understand which historical workflow state is incompatible:

- workflow name,
- schema/event name,
- instance key,
- journal height.

This makes semantic corruption diagnosable instead of looking like generic hosted startup failure.

## What Is Still Missing

The main missing protection is a hard manifest-consistency invariant.

We still need startup/snapshot logic to prove that:

- baseline state at height `H`,
- snapshot record at height `H`,
- and the manifest obtained from journaled manifest records up to `H`

all agree.

Without that, an operator or repair tool can still create a world that boots but is not replay-faithful.

## Required Invariant

For any promoted baseline or indexed replay snapshot used as an authoritative replay seed:

1. `snapshot.height = H`
2. `snapshot.manifest_hash = M`
3. replaying manifest journal records through height `H` must also yield `M`

If not, the world must not silently proceed as healthy.

Expected behavior:

- normal startup should fail closed with a typed `manifest_baseline_mismatch`-style error,
- recovery must use an explicit repair flow,
- repair must derive the corrected manifest from journal history rather than accepting an arbitrary replacement hash.

## Proposed Work

### A) Make Manifest Hash Fully Derived At Snapshot Time

When writing or promoting a snapshot:

- carry the manifest hash that is actually active at that replay height,
- persist it as part of the snapshot record,
- treat later mutation of that field as invalid.

This should be a derived replay result, not operator-editable state.

### B) Validate On Open

During hosted open:

- read the replay seed snapshot,
- derive or validate the manifest lineage through the seed height,
- reject the seed if the snapshot manifest does not match journal truth.

This should happen before live workflow replay continues.

### C) Fail Closed On Baseline Promotion

Promotion to `active_baseline` should require:

- valid snapshot metadata,
- valid receipt horizon semantics,
- manifest consistency with journal history at the same height.

Promotion should not be allowed on "bootable but unproven" snapshots.

### D) Repair Flow Must Be Journal-Derived

If a world is already inconsistent:

- repair tooling must compute the manifest at height `H` from journal history,
- then rebuild or reindex the snapshot/baseline against that manifest,
- not simply stamp a replacement manifest hash into runtime state.

### E) Add Regression Coverage

We should add an explicit corruption test:

1. create a world,
2. journal a manifest change,
3. write or mutate a snapshot/baseline with a different manifest hash,
4. assert hosted open fails with a precise typed error.

This should live next to the FDB hosted persistence/runtime tests.

## Operational Guidance Until This Lands

Until the invariant is enforced directly:

- do not treat "the world boots" as proof that hosted repair was correct,
- prefer rebuilding a fresh world from correct local source rather than force-repairing an old poisoned world,
- when a world has historical replay anomalies, validate:
  - active baseline height,
  - latest snapshot height,
  - manifest hash at runtime,
  - manifest hash in the journal lineage.

If those disagree, the world should be treated as semantically compromised even if control-plane reads still work.

## Exit Criteria

This gap is closed when:

- [ ] hosted startup rejects snapshot/baseline manifest mismatch explicitly,
- [ ] baseline promotion requires manifest-lineage consistency,
- [ ] repair tooling derives manifest truth from journal history,
- [ ] a targeted regression test covers this failure mode,
- [ ] operator docs state that bootability is not sufficient proof of consistency.
